# 開発ログ

## 1. AIをどう活用したか

今回の開発では、AIを**実装補助・壁打ち相手・調査補助**として活用しました。

### 活用方針

- 自力でも調べながら実装できるが、時間のかかる定型コード生成をAIに任せる
- UIコンポーネントのたたき台作成
- テストコードのベース作成
- GitHub Actions / CI/CD設定の相談
- TypeScriptエラーの原因調査
- 設計方針の壁打ち
- READMEやmdファイルの要約

### 意識したこと

- AIが生成したコードはそのまま使わず、必ず自分の目で確認する
- なぜそのコードになるのか理解してから採用する
- AIの回答を鵜呑みにせず、公式ドキュメントや実際の挙動も確認する
- 実装速度を上げる手段として使い、判断は自分で行う
- 文章を生成させる際書きたいことをできるだけ詳細に伝え、要約させる・必ず自分の目で確認する

### 学び

AIは非常に便利でしたが、**最終的な品質担保は開発者自身が行う必要がある**と感じました。
特に認証・権限制御・CI/CDなど重要部分ほど、自分で理解することが重要だと学びました。

## 2. 詰まった点とその解決方法

## GitHub Actions / CI/CD構築

今回初めて本格的にCI/CDを導入したため、最も苦戦しました。

### 詰まった点

- PR時とmainマージ後でテスト実行タイミングの違いが分からなかった
- E2EテストがGitHub Actions上で失敗する
- Secrets設定不足でログインテストが通らない
- Branch Protection Rulesの設定が複雑だった
- Required checks名と実際のJob名が一致していなかった

### 解決方法

- PR時にUnit Test / E2E Testが走る構成へ変更
- Repository SecretsにE2Eログイン用アカウント情報を設定
- `.env` をGitHub Actions内で生成する形へ変更
- Required checksを `test` Job名に合わせて再設定
- mainブランチ直接push禁止 + PR必須に変更

### 結果

- テスト成功時のみマージ可能
- main反映後にVercel自動デプロイ
- 品質を担保した状態で本番反映できる環境を構築できました

---

## デプロイエラー

### 詰まった点

- vercelにデプロイした際500エラーになる

### 原因

- vercelのFramework presetをOtherに設定していたこと

### 解決方法

- OtherからNextに変更

### 結果

デプロイできるようになり、サーバーで動作可能になりました

---

## E2Eテスト導入

### 詰まった点

- ローカルでは成功するがCI環境で失敗する
- ログイン後に `/reports` へ遷移しない

### 原因

- サーバー環境に.envがない

### 解決方法

- Playwright Traceで失敗原因を確認
- テストファイルで.envを作成

### 結果

E2Eが動くようになりCI/CDが完成しました

## 3. 技術選定の理由

## Next.js + React

- 現在主流のフロントエンド技術で情報量が多い
- App Routerなど新しい設計思想を学べる
- 実務でも採用例が多い

## TypeScript

- 型安全に開発したかったため
- 規模が大きくなっても保守しやすい
- 実務で必要とされることが多い

## Supabase

- 認証 / DB / RLSまで一体で扱える
- 個人開発でもスピード感を持って作れる

## Tailwind CSS

- 開発速度が速い
- デザイン調整がしやすい
- コンポーネント設計と相性が良い

## GitHub Actions

- 無料でCI/CDを構築しやすい
- GitHubとの連携が強い
- 実務でもよく使われるため学習価値が高い

## Vercel

- Next.jsとの相性が非常に良い
- デプロイが簡単
- Preview環境も使いやすい

### 総括

「最近よく使われている技術」「今後も需要がありそうな技術」「学んでいて楽しい技術」だと思ったので選定しました。

## 4. 設計判断の理由

今回の開発では、**限られた期間の中で完成度と学習効果のバランスを取ること**を重視しました。

## MVP思考で優先順位を決定

最初から機能を増やしすぎず、まず以下を優先しました。

- 認証
- チーム参加機能
- 日報CRUD
- コメント機能

まずコア機能を完成させ、その後に改善機能を追加する流れにしました。

## BaaS活用による開発効率化

認証・DB・権限制御をすべて自作せず、Supabaseを採用することで、

- 実装スピード向上
- セキュリティ担保
- RLS学習

を同時に実現しました。

## 型安全を早期に導入

後から型を整えるとコストが高いため、開発途中で strict mode を導入し、早い段階で品質改善を行いました。

## UIより機能完成を優先

期限内に成果物を完成させることを優先し、まずは使える状態を作った上で、後からUX改善・ダークモード・ローディング改善などを追加しました。

## 総括

理想的な設計よりも、

- 期限内に完成させる
- 実務に近い構成で作る
- 学びが大きい選択をする

この3点を重視して判断しました。
